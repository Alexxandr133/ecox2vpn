## VPN/VLESS сервис с Telegram‑ботом раздачи ключей

### Цель

Сделать самоуправляемый VPN‑сервис на VLESS (Xray/V2Ray), где:

- Telegram‑бот:
  - принимает команды от админа/пользователей,
  - создаёт/выдаёт VLESS‑ключи,
  - показывает статус и статистику.
- Бэкенд:
  - управляет Xray/V2Ray (создаёт юзеров, удаляет, продлевает),
  - хранит данные о клиентах (логин, тариф, срок, трафик),
  - ведёт учёт использованного трафика и сроков.

---

### Архитектура (в общих чертах)

- **Сервер 1 (VPN‑нод)**:
  - Ubuntu 24.04
  - Xray/V2Ray с протоколом **VLESS + TLS** (через nginx или напрямую),
  - один или несколько listen‑портов (443/80/порт под gRPC/WebSocket).
- **Сервер 2 (управляющий / бот)** или тот же:
  - Python/NodeJS/Go‑приложение:
    - Telegram‑бот (Bot API),
    - API/служба управления конфигом Xray/V2Ray,
    - периодический воркер (cron) для отключения просроченных аккаунтов.
  - База данных:
    - на старте можно SQLite/PostgreSQL,
    - таблицы: пользователи, подписки, устройства/ключи, лог трафика (агрегированный).

Все компоненты можно держать на одном VPS для начала.

---

### Протокол и сервер VPN

- Использовать **Xray** с VLESS:
  - VLESS + TCP + TLS (через nginx или Caddy),
  - опционально VLESS + WS + TLS (под доменом).
- Базовая настройка:
  - домен с Cloudflare/другим DNS,
  - `example.com` → IP сервера VPN,
  - автоматический сертификат через `acme.sh` или `certbot`.
- Xray‑конфиг:
  - `inbounds`:
    - порт 443, протокол VLESS, `decryption: none`,
    - `clients` — динамически управляемый список юзеров (UUID, email).
  - `outbounds`: обычный `freedom` (выход в интернет).

---

### Telegram‑бот: функционал

Минимальный функционал:

- **/start** — приветствие, инфо о сервисе.
- **/buy /get /new** — создание нового VLESS‑ключа:
  - генерируем UUID,
  - создаём запись в БД (пользователь, срок, лимит),
  - добавляем клиента в конфиг Xray,
  - возвращаем пользователю VLESS‑URI и готовые ссылки/конфиги.
- **/my** — показать активные ключи пользователя:
  - срок действия,
  - примерный использованный трафик (если считаем).
- **/renew** — продление ключа.

Админ‑команды:

- **/admin_users** — список/поиск пользователей.
- **/admin_add** — выдать ключ конкретному пользователю.
- **/admin_ban /admin_off** — отключить ключ/юзера.

Механика общения:

- Bot API через long polling (для простоты) или webhook.
- Авторизация админов по ID чатов (whitelist ID).

---

### Управление Xray/V2Ray

Варианты:

- **Через конфиг‑файл и перезапуск** (простой старт):
  - бот редактирует JSON‑конфиг Xray (inbound.clients),
  - делает `systemctl reload xray` или перезапуск,
  - хранит «истину» в БД и каждый раз генерирует конфиг из БД.
- **Через встроенный API (dynamic)** — если хотим без перезапусков:
  - включить `API` inbound/outbound в Xray,
  - бот ходит по gRPC/JSON‑API Xray и добавляет/удаляет клиентов на лету.

Для первой версии проще сделать **генерацию конфига из БД + reload**.

### Модель данных (черновик)

- **users**:
  - `id` (int),
  - `tg_id` (Telegram user id),
  - `username` (nullable),
  - `role` (`user` / `admin`),
  - `created_at`.
- **subscriptions**:
  - `id`,
  - `user_id` (FK → users),
  - `uuid` (VLESS UUID),
  - `server_id` или tag сервера,
  - `plan` (имя тарифа),
  - `expires_at`,
  - `traffic_limit_mb` (nullable, если безлимит),
  - `traffic_used_mb` (агрегируется).
- **servers** (если несколько нод):
  - `id`,
  - `name`,
  - `host`,
  - `port`,
  - `tls_domain`,
  - дополнительные параметры.

---

### Учёт трафика и отключение

Варианты счёта:

- **Лог‑анализ Xray**:
  - включить лог запросов/сессий,
  - периодически парсить лог, аггрегировать по UUID,
  - обновлять `traffic_used_mb` в БД.
- **Внешний скрипт/плагин**:
  - если используем готовые решения для учёта.

Отключение:

- cron‑джоб/планировщик:
  - раз в N минут:
    - находит подписки с `expires_at < now()` или `traffic_used_mb > limit`,
    - удаляет/отключает эти UUID из конфига Xray,
    - помечает в БД как `disabled`,
    - отправляет уведомление в Telegram.

---

### Безопасность и деплой

- **Разные пользователи на сервере**:
  - Xray и бот под отдельным `system`‑пользователем, без root там, где не нужно.
- **VPN‑нод**:
  - открыть только нужные порты (80/443 и API, если нужно, но лучше только локально),
  - `ufw`/firewalld для ограничения.
- **Бот/бэкенд**:
  - деплой через `systemd` или Docker‑compose,
  - хранить токен бота и доступы к БД в `.env`/секретах.

---

### Минимальный первый релиз (MVP)

1. Один сервер (Xray + бот на том же VPS).
2. Xray с VLESS + TLS по домену.
3. SQLite/Postgres + миграции.
4. Бот:
   - `/start`, `/new`, `/my`,
   - базовый `/admin_*` для ручного управления.
5. Генерация конфига Xray из БД + `systemctl reload xray`.

Дальше можно:

- добавлять биллинг/оплаты,
- несколько нод (балансировка),
- красивый веб‑интерфейс поверх бота.
